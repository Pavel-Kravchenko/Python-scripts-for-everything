Problem 1: Algorithm Identification 15 points

Identify which sorting algorithm produces each sequence of steps when sorting [5, 2, 8, 1, 9]:

a) (5 pts)

[5, 2, 8, 1, 9] → [2, 5, 8, 1, 9] → [2, 5, 8, 1, 9] → [1, 2, 5, 8, 9]
(element 2 inserted) (element 8 stays) (elements 1,5,8 shift)

b) (5 pts)

[5, 2, 8, 1, 9] → [1, 2, 8, 5, 9] → [1, 2, 8, 5, 9] → [1, 2, 5, 8, 9]
(find min=1, swap with 5) (min=2, already in place) (find min=5, swap with 8)

c) (5 pts)

[5, 2, 8, 1, 9] split into [5, 2] and [8, 1, 9]
[5, 2] → [2, 5], [8, 1, 9] → [1] and [8, 9] → [1, 8, 9]
merge [2, 5] and [1, 8, 9] → [1, 2, 5, 8, 9]

Answer Key

a) Insertion Sort - Each element is inserted into its correct position in the sorted portion.

b) Selection Sort - Repeatedly finds minimum element and places it at the beginning.

c) Merge Sort - Divides array in half recursively, then merges sorted halves.

Problem 4: Algorithm Selection 15 points

For each scenario, recommend the best sorting algorithm and justify your choice:

a) (5 pts)

Sorting 10 million 32-bit integers that are mostly unique.

b) (5 pts)

Sorting a deck of 52 playing cards by suit, then by rank within each suit.

c) (5 pts)

Real-time sorting of incoming network packets where packets arrive nearly in order.

Answer Key

a) Radix Sort or Quick Sort

  • Radix Sort: O(32n) = O(n) for 32-bit integers. Very fast for large n.
  • Quick Sort: O(n log n) average, but good cache performance.
  • Not Counting Sort: range too large (2³² values)

b) Radix Sort (LSD) with two passes

  • First pass: stable sort by rank (13 values)
  • Second pass: stable sort by suit (4 values)
  • Stability ensures rank order preserved within suits
  • Also acceptable: Two stable sorts (any stable algorithm)

c) Insertion Sort

  • O(n) for nearly sorted data
  • Adaptive: performance improves with sorted input
  • Simple, low overhead for real-time systems
  • Online: can sort as elements arrive

Problem 5: Merge Sort Implementation 20 points

The following merge function has bugs. Find and fix them:

def merge(left, right):
    result = []
    i = j = 0
    
    while i < len(left) and j < len(right):
        if left[i] < right[j]:      # Bug 1?
            result.append(left[i])
            i += 1
        else:
            result.append(right[i])  # Bug 2?
            i += 1                   # Bug 3?
    
    result.extend(left[i:])
    result.extend(right[i:])         # Bug 4?
    
    return result

Answer Key

Bug 1 (line 6): Should be <= to maintain stability

if left[i] <= right[j]:  # Stable: prefer left on ties

Bug 2 (line 10): Should access right[j], not right[i]

result.append(right[j])  # Access right array with j

Bug 3 (line 11): Should increment j, not i

j += 1  # Move right pointer

Bug 4 (line 14): Should use right[j:], not right[i:]

result.extend(right[j:])  # Use j for right array

Corrected Code:

def merge(left, right):
    result = []
    i = j = 0
    
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:     # Fixed: <= for stability
            result.append(left[i])
            i += 1
        else:
            result.append(right[j]) # Fixed: right[j]
            j += 1                  # Fixed: j += 1
    
    result.extend(left[i:])
    result.extend(right[j:])        # Fixed: right[j:]
    
    return result

Bonus: In-Place Stable Sort 10 points (extra credit)

Explain why it's difficult to create a sorting algorithm that is simultaneously:

  • O(n log n) worst-case time
  • O(1) space (in-place)
  • Stable

Which algorithms come closest? What trade-offs do they make?

Answer Key

The Trilemma: These three properties are difficult to achieve together because:

  • Merge Sort: O(n log n), stable, but O(n) space. The merge step needs auxiliary space to maintain stability.
  • Heap Sort: O(n log n), O(1) space, but NOT stable. The heap operations can reorder equal elements.
  • Quick Sort: O(n log n) average (O(n²) worst), O(log n) space, NOT stable. Partitioning disrupts relative order.

Block Sort (WikiSort): Comes closest! It achieves all three properties through complex block manipulation techniques, but has high constant factors and implementation complexity.

Why it's hard:

  • Stability requires remembering original positions
  • In-place operation limits how we can move elements
  • Achieving both usually requires O(n) extra space for indices
  • In-place merge (for in-place merge sort) can be done but is complex and slow

Full credit for explaining the trade-offs and mentioning at least 2 algorithms.
