Queue Implementation

(a) [10 pts] Implement a circular queue using an array of size 5. Show the state of the array, front, and rear pointers after each operation:

enqueue(1), enqueue(2), enqueue(3), dequeue(), dequeue(), enqueue(4), enqueue(5), enqueue(6), dequeue()

(b) [10 pts] Implement isEmpty() and isFull() for a circular queue. What is the maximum number of elements that can be stored in a circular queue of size n? Explain.

Answer Key:

(a) (Array indices 0-4, front=F, rear=R)

Initial: [ _ | _ | _ | _ | _ ] F=0, R=0 enqueue(1): [ 1 | _ | _ | _ | _ ] F=0, R=1 enqueue(2): [ 1 | 2 | _ | _ | _ ] F=0, R=2 enqueue(3): [ 1 | 2 | 3 | _ | _ ] F=0, R=3 dequeue(): [ _ | 2 | 3 | _ | _ ] F=1, R=3 → returns 1 dequeue(): [ _ | _ | 3 | _ | _ ] F=2, R=3 → returns 2 enqueue(4): [ _ | _ | 3 | 4 | _ ] F=2, R=4 enqueue(5): [ _ | _ | 3 | 4 | 5 ] F=2, R=0 (wrap!) enqueue(6): [ 6 | _ | 3 | 4 | 5 ] F=2, R=1 dequeue(): [ 6 | _ | _ | 4 | 5 ] F=3, R=1 → returns 3

(b)

def isEmpty(): return front == rear def isFull(): return (rear + 1) % size == front

Maximum elements: n-1

We sacrifice one slot to distinguish empty from full. If we allowed n elements, front==rear would be ambiguous (could mean empty or full).

Problem 4 20 points

Hash Table Analysis

(a) [8 pts] Insert keys 10, 22, 31, 4, 15, 28, 17, 88 into a hash table of size 11 using h(k) = k mod 11 with:

  1. Separate Chaining
  2. Linear Probing

Draw both final hash tables.

(b) [6 pts] What is the load factor after all insertions? What happens to performance as load factor increases for each method?

(c) [6 pts] Why do we prefer prime numbers for hash table sizes? Give an example showing how a non-prime size can cause clustering.

Answer Key:

(a) Hash values: 10→10, 22→0, 31→9, 4→4, 15→4, 28→6, 17→6, 88→0

Separate Chaining:

[0]: 22 → 88 [1]: empty [2]: empty [3]: empty [4]: 4 → 15 [5]: empty [6]: 28 → 17 [7]: empty [8]: empty [9]: 31 [10]: 10

Linear Probing:

[0]: 22 [1]: 88 (probed from 0) [2]: empty [3]: empty [4]: 4 [5]: 15 (probed from 4) [6]: 28 [7]: 17 (probed from 6) [8]: empty [9]: 31 [10]: 10

(b) Load factor = 8/11 ≈ 0.73

  • Chaining: Graceful degradation. Average chain length increases linearly. Can exceed α > 1.
  • Linear Probing: Rapid degradation. Clustering gets worse. Performance degrades sharply as α → 1. Cannot exceed α = 1.

(c) Prime sizes distribute keys more uniformly.

Example: If size = 10 and keys are multiples of 5 (5, 10, 15, 20, ...): All hash to indices 0 or 5 only → 80% of table unused!

With size = 11 (prime): 5→5, 10→10, 15→4, 20→9, 25→3 → better distribution

Problem 5 15 points

Cycle Detection

(a) [8 pts] Prove that Floyd's cycle detection algorithm is correct. Why must the slow and fast pointers meet if there's a cycle?

(b) [7 pts] After detecting a cycle, how would you find the start of the cycle? Explain the algorithm and its time complexity.

Answer Key:

(a) Proof:

  • Let the cycle have length C
  • Once both pointers are in the cycle, their relative distance decreases by 1 each step (fast gains 1 on slow)
  • Since distance is finite and decreasing, they must eventually meet
  • Maximum steps before meeting: O(n) where n is list length

(b) Finding cycle start:

1. Detect meeting point using Floyd's 2. Reset one pointer to head 3. Move both pointers one step at a time 4. They meet at the cycle start

Why it works: Let distance from head to cycle start = m, and meeting point is k steps into the cycle. The fast pointer traveled 2(m+k) while slow traveled (m+k). The difference m+k must be a multiple of cycle length. When slow is reset to head and both move at same speed, they meet at the start after m steps.

Time complexity: O(n)

Problem 6 15 points

Design Problem: Browser History

Design a data structure to implement browser back/forward functionality:

  • visit(url) - Visit a new URL (clears forward history)
  • back(steps) - Go back up to 'steps' pages
  • forward(steps) - Go forward up to 'steps' pages

(a) [7 pts] What data structure(s) would you use? Justify your choice.

(b) [8 pts] Implement the three operations with optimal time complexity. State the complexity of each operation.

Answer Key:

(a) Use two stacks: back_stack and forward_stack, plus current_url.

  • back_stack: pages we can go back to
  • forward_stack: pages we can go forward to
  • Stacks are ideal because we access history in LIFO order

(b)

class BrowserHistory: def __init__(self, homepage): self.current = homepage self.back_stack = [] self.forward_stack = [] def visit(self, url): # O(1) self.back_stack.append(self.current) self.current = url self.forward_stack.clear() # Clear forward history def back(self, steps): # O(steps) while steps > 0 and self.back_stack: self.forward_stack.append(self.current) self.current = self.back_stack.pop() steps -= 1 return self.current def forward(self, steps): # O(steps) while steps > 0 and self.forward_stack: self.back_stack.append(self.current) self.current = self.forward_stack.pop() steps -= 1 return self.current

Complexities: visit O(1), back O(steps), forward O(steps)

Bonus +10 points

Min Stack

Design a stack that supports push, pop, top, and getMin, all in O(1) time.

Hint: What additional information can you store with each element?

Answer Key:

Store pairs of (value, min_so_far) on the stack:

class MinStack: def __init__(self): self.stack = [] # stores (value, min_so_far) def push(self, val): # O(1) if not self.stack: self.stack.append((val, val)) else: current_min = min(val, self.stack[-1][1]) self.stack.append((val, current_min)) def pop(self): # O(1) return self.stack.pop()[0] def top(self): # O(1) return self.stack[-1][0] def getMin(self): # O(1) return self.stack[-1][1]

Each element "remembers" the minimum of all elements below it (including itself).
