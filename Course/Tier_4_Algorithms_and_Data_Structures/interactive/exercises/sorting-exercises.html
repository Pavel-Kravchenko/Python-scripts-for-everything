Exercise 1: Trace Bubble Sort

Given the array [64, 34, 25, 12, 22, 11, 90], trace through Bubble Sort.

Question: After the first complete pass, what is the state of the array?

✅ Solution

After the first pass of Bubble Sort:

Initial: [64, 34, 25, 12, 22, 11, 90]

Compare 64 & 34 → swap  → [34, 64, 25, 12, 22, 11, 90]
Compare 64 & 25 → swap  → [34, 25, 64, 12, 22, 11, 90]
Compare 64 & 12 → swap  → [34, 25, 12, 64, 22, 11, 90]
Compare 64 & 22 → swap  → [34, 25, 12, 22, 64, 11, 90]
Compare 64 & 11 → swap  → [34, 25, 12, 22, 11, 64, 90]
Compare 64 & 90 → no swap

After first pass: [34, 25, 12, 22, 11, 64, 90]

Key insight: 90 (the largest element) has "bubbled up" to its final position at the end.

Exercise 2: Complexity Analysis

Match each scenario with the most appropriate sorting algorithm:

  1. Sorting 1 million integers between 0-999
  2. Sorting a nearly sorted array of 10,000 elements
  3. Sorting linked list nodes
  4. Need guaranteed O(n log n) with stability

✅ Solution

  1. Counting Sort - Perfect for limited range integers. O(n + k) where k = 1000. Much faster than comparison sorts for this case.
  2. Insertion Sort - O(n) for nearly sorted data due to its adaptive nature. Each element moves only a few positions.
  3. Merge Sort - Ideal for linked lists because it doesn't require random access. Can be done in O(1) space for linked lists.
  4. Merge Sort - Always O(n log n) regardless of input, and stable. QuickSort has O(n²) worst case and isn't stable.

Exercise 4: Merge Sort Recursion

For the array [38, 27, 43, 3, 9, 82, 10]:

Questions:

  1. How many times will the array be split?
  2. What are all the merge operations in order?

✅ Solution

Splitting (Divide):
[38, 27, 43, 3, 9, 82, 10]
      ↓

    ↓        ↓           ↓

1. [38] + [27] → [27, 38]

Exercise 5: Stability Concept

Consider sorting these students by score:

[("Alice", 85), ("Bob", 90), ("Carol", 85), ("Dave", 90)]

Questions:

  1. What does it mean for a sort to be "stable"?
  2. If we use a stable sort, what's the guaranteed output?
  3. Which common sorts are stable? Which are not?

✅ Solution

1. Stability Definition:

A sorting algorithm is stable if elements with equal keys maintain their relative order from the input.

2. Stable Sort Output:

[("Alice", 85), ("Carol", 85), ("Bob", 90), ("Dave", 90)]
      ↑              ↑             ↑            ↑
  Alice before Carol (original order preserved)
                       Bob before Dave (original order preserved)

3. Stability by Algorithm:

Stable ✅ Unstable ❌
Bubble Sort QuickSort
Insertion Sort Heap Sort
Merge Sort Selection Sort
Counting Sort Shell Sort

Exercise 6: Best/Worst Case Analysis

For each algorithm, describe the input that causes best and worst case:

  1. Bubble Sort
  2. QuickSort (with last element as pivot)
  3. Insertion Sort

✅ Solution

1. Bubble Sort
  • Best Case O(n): Already sorted array [1, 2, 3, 4, 5]. First pass detects no swaps and terminates.
  • Worst Case O(n²): Reverse sorted [5, 4, 3, 2, 1]. Every element must bubble to the end.
2. QuickSort
  • Best Case O(n log n): Pivot always splits array in half. Example: median element happens to be at the end.
  • Worst Case O(n²): Already sorted [1, 2, 3, 4, 5] or reverse sorted. Pivot is always min/max, creating n partitions of size n-1, n-2, etc.
3. Insertion Sort
  • Best Case O(n): Already sorted. Each element compared once with its predecessor, no shifts needed.
  • Worst Case O(n²): Reverse sorted. Each element must shift past all previous elements.

Exercise 7: Coding Challenge

Implement: Write a function that determines if an array can be sorted with exactly one swap.

def can_sort_one_swap(arr):
    # Return True if exactly one swap sorts the array
    # Return False otherwise
    pass

# Examples:
# can_sort_one_swap([1, 5, 3, 4, 2, 6]) → True (swap 5 and 2)
# can_sort_one_swap([1, 2, 3, 4, 5]) → False (already sorted, 0 swaps)
# can_sort_one_swap([3, 2, 1]) → False (needs more than 1 swap)

✅ Solution

def can_sort_one_swap(arr):
    # Find positions where arr differs from sorted version
    sorted_arr = sorted(arr)
    diffs = [i for i in range(len(arr)) if arr[i] != sorted_arr[i]]
    
    # Need exactly 2 positions that differ
    if len(diffs) != 2:
        return False
    
    # Check if swapping those positions sorts the array
    i, j = diffs
    arr[i], arr[j] = arr[j], arr[i]
    return arr == sorted_arr

# Alternative approach without sorting:
def can_sort_one_swap_v2(arr):
    n = len(arr)
    # Find first position where arr[i] > arr[i+1]
    first = -1
    for i in range(n - 1):
        if arr[i] > arr[i + 1]:
            first = i
            break
    
    if first == -1:
        return False  # Already sorted
    
    # Find position to swap with
    for j in range(n - 1, first, -1):
        if arr[j] < arr[first]:
            # Try swap
            arr[first], arr[j] = arr[j], arr[first]
            # Check if sorted
            is_sorted = all(arr[k] <= arr[k+1] for k in range(n-1))
            arr[first], arr[j] = arr[j], arr[first]  # Swap back
            if is_sorted:
                return True
    
    return False

Exercise 8: Space Complexity

Explain why:

  1. Merge Sort requires O(n) extra space
  2. QuickSort requires O(log n) space
  3. In-place Merge Sort is possible but rarely used

✅ Solution

1. Merge Sort O(n) Space

The merge step requires a temporary array to hold elements while merging two sorted subarrays. At the top level, we merge two n/2 sized arrays, needing O(n) space.

2. QuickSort O(log n) Space

QuickSort sorts in-place but uses the call stack for recursion. With balanced partitions, recursion depth is O(log n). With tail recursion optimization (processing smaller partition first), we can guarantee O(log n) stack space even in worst case.

3. In-place Merge Sort

It's possible to merge in-place using techniques like "block merge" or rotation-based merging, but:

  • Constant factors become very high
  • Implementation is complex
  • Cache performance suffers
  • The O(n) space of standard merge sort is usually acceptable
